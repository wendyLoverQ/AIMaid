# AIMaid 项目开发规则

本文件适用于 `AIMaid`。修改代码前必须先阅读并遵守。

## 一、核心原则

1. `AIMaid` 是当前正式项目：Electron/React 前端 + C# CoreHost。
2. `AI_maid` 是旧 WPF 项目，只作为业务和数据结构参考；除非任务明确要求，不得修改旧项目。
3. 不把项目当 Demo，不写一次性补丁，不为了“先跑起来”破坏正式结构。
4. 不猜。看不懂、拿不准或资料可能过期时，先查当前代码、日志、官方文档、源码或 Issue。
5. 严禁兜底降级、吞异常、静默跳过、伪造成功、默认成功或用延迟/重试掩盖问题。
6. 不扩大任务范围，不顺手重构无关代码，不重复建立已有能力。
7. 项目只使用当前固定视觉主题，禁止新增深色/浅色切换、跟随系统、自动换肤或主题架构。
8. ASR 只接免费的第三方 API，禁止引入或部署 Whisper、FunASR 等本地 ASR 模型。
9. 并行任务不得修改公共版本文件；由同批次最终集成任务统一更新版本号并提交主分支。

## 二、开发流程

1. 写代码前先给出简短方案：改什么、怎么改、影响哪里。
2. 需求不明确或存在多种业务实现时，先给主人选择；未确认前不得开发。
3. 已经出现的问题，先复现并查日志、数据和调用链，确认根因后再改。
4. 开发中发现原方案不成立、需求冲突或新的关键问题时，立即暂停并说明，不得擅自换方案。
5. 修改公共接口、协议、DTO、配置或数据结构时，必须同步检查全部调用方。
6. 完成后只说明实际修改和真实验证结果，不夸大，不生成无关文档。

## 三、业务承接与调用链

1. 修复或迁移旧业务时，先对照旧 `AI_maid` 的真实行为，明确：保留、重做或废弃。
2. 不得仅凭“Core 有路由”或“页面有按钮”判断功能已经完成。
3. 业务修改必须检查完整链路：

   `Renderer 页面 → Bridge/Preload 权限 → IPC Router → Core 契约与 Handler → 数据库/外部适配器 → 打包产物`

4. 业务编排和真实状态放在 C# Core；React 只保存输入草稿、筛选、Tab 和弹窗等临时 UI 状态。
5. TypeScript 契约、C# RequestTypes、Handler、窗口权限和调用代码必须保持一致。
6. 旧数据库兼容层不得静默忽略关键字段或用空值补出“成功”；新增映射必须验证真实旧库的读、写、改、删。
7. 文档与代码不一致时，以当前代码为准，并在同次修改中更新相关文档。

## 四、Codex 执行边界

1. Codex 只实施主人已经确认的明确修改，不负责自行盘点项目、核对旧业务或寻找下一项任务。
2. 问题定位、业务对照、调用链核对和测试方案由主人或外部审查者提供；Codex 不得重复调查。
3. 除非任务明确指定，不新增测试、不运行全量测试、不扩大检查范围。
4. 修改完成后只执行任务中明确要求的最小构建或命令；未指定时不得自行增加验证步骤。
5. 发布目录唯一为：
   `C:\Users\49213\Desktop\A\codex\AIMaid\apps\desktop\release`
6. 每次完成修改后提交并推送 GitHub。

## 五、版本号

1. 版本格式：`YYYY.M.D.DailyIndex.TotalCount`，日期按 `Asia/Shanghai`。
2. `InformationalVersion` 使用完整五段；`AssemblyVersion` 和 `FileVersion` 使用 `YYYY.M.D.TotalCount`。
3. 版本字段统一维护在根目录 `Directory.Build.props`，禁止在业务代码或脚本中硬编码。
4. 每次发布同时更新三个版本字段；`DailyIndex` 每日重置，`TotalCount` 永久递增。
5. 版本号修改只属于发布元数据，不得夹带无关业务改动。

## 六、数据库报告

1. 修改 Schema、数据库映射或迁移目标结构后，必须运行 Schema 基线检查。
2. 普通 UI 或业务提交不运行数据库报告；真实数据库快照只在主人明确要求时更新。
3. 原始数据库禁止提交；报告差异必须真实处理，不得直接覆盖基线掩盖问题。

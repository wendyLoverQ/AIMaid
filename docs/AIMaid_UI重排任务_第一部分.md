# AIMaid APP UI 重排任务（第一部分：核心交互与配置中心）

## 1. 本任务的定位

本文件是整套 AIMaid APP UI 重排工作的第一半，负责用户最高频的入口、对话、角色、提醒、笔记、设置与状态页面。

本任务不是恢复旧 WPF 外观，也不是把旧项目页面逐像素搬到 React。旧项目 `C:\Users\49213\Desktop\A\codex\AI_maid` 只用于核对已有功能、字段、操作路径、窗口用途和边界条件；新版必须在不改变功能的前提下重新建立清楚、现代、适合桌面端的视觉层级。

与 `AIMaid_UI重排任务_第二部分.md` 合并后，应覆盖当前 APP 的全部业务窗口。两部分可以并行实施，但必须共用现有 Design Token、`components/ui.ts` 组件出口和同一套页面骨架规则。

## 2. 绝对约束

### 2.1 只允许修改

- React 页面结构中的纯展示层级与排版顺序。
- `components/ui` 内已有控件的组合方式。
- 与本部分页面直接相关的 CSS、间距、宽高、网格、滚动、视觉层级和响应式规则。
- 文案的视觉分组、辅助说明位置与可访问性标签；不得改变操作语义。
- 必要的纯 UI 组件抽取，但不得在页面内重新制造第二套基础控件。

### 2.2 严禁修改

- C# Core、数据库、IPC、preload、主进程、窗口注册、窗口类型和窗口打开/关闭行为。
- `bridge` 方法、Core request type、payload、返回结构、事件名、超时、业务判断、保存时机和错误语义。
- 聊天、提醒、角色卡、TTS、笔记、设置、状态采集等任何业务逻辑。
- 为了适配新 UI 增加模拟数据、假状态、降级逻辑或兜底分支。
- 删除、隐藏或合并现有功能入口。
- 把旧项目的 WPF/WebView 布局原样照抄到 React。
- 新增“深色模式”“浅色模式”“跟随系统”模式。主题选择入口必须保留，主题均为亮色系配色方案。
- 使用页面私有裸 `button/input/select/textarea` 或 iframe 代替统一控件。

## 4. 全局重排方向

### 4.1 新版视觉原则

- 以“桌面工作台”而不是“表单合集”为视觉模型。
- 一个窗口只保留一个最强主标题、一个主操作区和一条清晰阅读路径。
- 大页面使用固定标题栏 + 可滚动内容区；左右栏各自滚动时必须清楚区分滚动容器。
- 操作按频率与风险分级：主操作、普通操作、低频操作、危险操作不能同权排列。
- 信息卡片用于表达真正的分组，不允许每一行都套卡片。
- 不用大面积空白弥补结构问题；使用内容最大宽度、列宽和密度控制阅读节奏。
- 默认桌面宽度优先，同时保证窗口缩窄后不出现横向裁切、按钮压字和表单脱离阅读顺序。
- 使用真实图标库图标；不使用 emoji、文本符号或手绘 SVG 充当正式图标。

### 4.2 统一页面骨架

- 顶部：`WindowTitleBar`，展示窗口名称、简短用途和最高频动作。
- 内容入口：需要搜索或筛选的页面使用 `PageToolbar/Toolbar`，不能把搜索框孤零零放进无语义容器。
- 主体：优先使用 `Page + PageContent + WorkspaceGrid/Stack/Surface`。
- 详情编辑：桌面宽屏采用“列表/导航 + 工作区”，窄屏降为单列并保证当前内容优先。
- 空、加载、错误、离线和未授权状态必须占据对应内容区，而不是挤在页面角落。
- 弹层统一使用 `Dialog/ConfirmDialog/Drawer/Popover/ContextMenu`，焦点、Esc、危险操作位置保持一致。

### 4.3 响应式基线

- 宽度 `>= 1040px`：允许双栏或三段工作区，但内容主列不得无限拉宽。
- 宽度 `720–1039px`：侧栏收窄，工具栏允许换行，表单降为一至两列。
- 宽度 `< 720px`：单列；列表与详情改为主从切换或纵向排列；底部主操作保持可见。
- 所有控件保持至少 36px 常规高度；图标按钮点击区不得小于 36×36px。
- 任何页面不得依赖固定像素宽度才能完成核心任务。

## 5. 页面任务

## 5.1 工作台 `MainPage`

### 当前问题

- 目前只是标题后平铺 9 个文字 `Pressable`，没有功能分类、图标、说明、主次和空间节奏。
- 高频功能与低频工具同权，用户不能快速建立入口记忆。
- 旧版入口视觉过于拥挤，但其“从统一入口进入各工具”的功能定位需要保留。

### 目标布局

- 顶部保留“工作台”，右侧仅放窗口级动作，不在标题区堆入口。
- 首屏使用“主要操作 + 分类工具区”：
  - 高频区：角色对话中心、提醒事项、记事本。
  - 媒体区：视频库、远程视频中心。
  - 工具区：BTC、计时、密码库、快捷脚本。
- 入口使用一致的功能卡片：真实图标、标题、单行用途说明、明确 hover/focus/pressed 状态。
- 卡片网格宽屏 3 列，中宽 2 列，窄屏 1 列；不得照抄旧版按钮矩阵。
- 保留 `bridge.window.open(target)` 后关闭工作台的现有行为。

### 验收

- 9 个入口全部存在且 target 不变。
- 3 秒内可分辨对话、提醒、笔记三个高频入口。
- 窄窗口无横向滚动，卡片文字不截断。

## 5.2 快捷输入 `PromptPage`

### 当前问题

- 视觉信息只剩标题、分隔空容器和单行文本框，快捷键语义不可见。
- 透明浮层窗口缺少输入焦点、发送方式与关闭方式的视觉反馈。
- 普通聊天、继续对话、TTS 试听、脚本命令和提醒识别由键盘修饰键触发，但界面没有轻量提示。

### 目标布局

- 保持紧凑悬浮命令面板，不改成完整聊天页。
- 面板分为标题提示、输入区、快捷键提示三层；输入区是绝对视觉焦点。
- 在底部以 `Keycap`/辅助文字提示：Enter 提交、Ctrl+Enter 继续对话、Shift+Enter TTS 试听、Esc 关闭。
- 可增加纯展示的输入类型提示，但不得改变现有解析、提交和窗口隐藏逻辑。
- 面板在失焦/背景点击时仍按现有行为关闭；透明背景和圆角边界清晰，不出现双层卡片。

### 验收

- `submit()`、提醒解析、脚本启动、聊天、TTS 路径零改动。
- 键盘四种操作均可从界面理解。
- 默认焦点、重新聚焦清空内容、背景点击关闭行为保持不变。

## 5.9 系统设置 `SettingsPage`

### 当前问题

- 快捷键页保持动作目录与录制逻辑，使用表格/列表行表达，不堆重复卡片。
- 高级/诊断默认使用 Accordion 分组，避免一次展示全部长文本区；展开不改变数据加载和保存行为。
- 小窗口下导航转为顶部 Combobox/Drawer，内容单列。

### 验收

- 所有现有设置项均可找到，分类和 Core 调用不变。
- 搜索可以视觉定位匹配设置；无结果有明确空状态。
- 保存中、成功、失败、需重启等状态靠近相关设置。
- 页面不出现嵌套滚动失控、横向滚动或操作按钮被推出视口。

- 页面与应用始终使用 `color-scheme: only light`。

## 5.11 状态面板 `StatusPage`

### 当前问题

- 角色、核心服务、云服务器、资源、网络和 Codex 额度大量信息连续铺陈，所有卡片视觉权重相近。
- 当前状态、历史延迟、容量值和错误详情缺少“先看是否正常，再看细节”的扫描路径。

### 目标布局

- 顶部提供整体健康摘要：正常/异常/未知数量、最近刷新时间、刷新动作。
- 第一层为核心服务与当前角色，第二层为服务器与额度，第三层为系统资源与网络明细。
- 在线/离线/未知使用 `StatusBadge + 文本`，不能只用圆点颜色。
- CPU/GPU/内存/配额使用一致进度表达，数值和单位对齐。
- 服务器指标允许宽屏双列，窄屏单列；错误详情可以展开，不长期占据主视图。
- 数据刷新不得导致卡片尺寸大幅跳动。

### 验收

- 当前页面所有真实指标完整保留：角色、TTS、模型、Live2D、腾讯云、AWS、Codex、CPU、GPU、内存、网络延迟。
- 刷新、加载、部分失败和无数据状态各自清楚。
- 状态含义可在不依赖颜色的情况下理解。

# AIMaid UI 重排任务书（第二部分：媒体、工具与系统侧）

> 本文是整个 UI 重排工作的第二个执行分片，不是第二套设计方案。执行者只负责本文列出的页面，并与《第一部分》在文件和页面边界上并行推进。

## 1. 任务目标

在不改变任何功能和业务行为的前提下，重新组织媒体、工具与系统侧页面的视觉层级、信息密度、对齐关系、滚动区域和响应式布局，使当前 React 应用从“控件按代码顺序平铺”变成清晰、稳定、可维护的正式桌面应用界面。

本任务必须同时满足以下目标：

1. 以当前 AIMaid React 页面为唯一实现基础，使用已有统一组件和 `theme/tokens.css`。
2. 旧项目 `C:\Users\49213\Desktop\A\codex\AI_maid` 仅用于核对功能、字段、交互、窗口尺寸及透明窗口约束，不复制旧 WPF/HTML 的布局和视觉。
3. 页面先建立稳定的“标题区—主操作区—工作区—状态区”层级，再处理局部美化；不能继续依靠无语义 `Container` 自然堆叠。
4. 同类操作位置一致：页面级操作放标题或工具栏，当前对象操作贴近对象，危险操作与普通操作分组，状态反馈进入固定状态区域。
5. 保留主题选择按钮和配色主题能力，但全项目不得出现“深色模式、浅色模式、跟随系统”三种外观模式或对应切换入口。
6. 所有页面在当前窗口最小尺寸、默认尺寸和放大尺寸下都不得遮挡、裁切、横向溢出或让主操作滚出可达范围。

## 2. 绝对禁止项

执行本任务时禁止：

- 修改业务逻辑、状态转换、Core 调用、IPC、协议、事件名、DTO、数据结构、持久化、定时器行为、播放行为、下载行为或外部网站行为。
- 修改 `window-registry.ts` 中的窗口类型、默认尺寸、最小尺寸、透明度、置顶、可调整大小、任务栏显示、关闭策略、焦点策略等窗口属性。
- 新增、删除或合并现有功能入口；不得把“重新排版”变成产品功能改版。
- 改变按钮、列表项、右键菜单、双击、键盘、拖动、粘贴、导入、播放、删除、确认等事件的触发条件和处理函数。
- 改变任何 Core 路由字符串、传参字段、超时、请求顺序或数据刷新策略。
- 改变抖音 `ExternalWebview` 的地址、partition、导航、登录、验证码、会话保存或关闭流程。
- 为缺失数据、图标、资源、接口或异常新增兜底、假数据、默认成功、静默替代或降级路径。
- 在业务页面中写裸 `button/input/select/textarea/dialog/table`、内联视觉样式、页面私有 CSS 或 `div onClick`。
- 复制旧项目的固定像素列宽、灰色面板套面板、WPF 控件外观、旧 HTML 卡片样式或旧截图布局。
- 新增深色、浅色、自动、跟随系统相关状态、枚举、按钮、菜单或媒体查询。`prefers-color-scheme` 不得参与应用配色。
- 借 UI 调整顺手改文案语义、权限提示、风险等级、错误信息、数据格式或业务字段名称。

## 5. 全局布局规则

### 5.1 页面骨架

普通可缩放模块窗口统一按以下层级组织：

1. `WindowTitleBar`：仅承载窗口标题、简短副标题和真正的窗口级操作。
2. 页面标题/上下文区：优先用 `PageHeader`，显示当前对象、数量、状态和少量主操作。
3. 页面工具栏：用 `Toolbar` 或 `PageToolbar`，搜索、筛选、排序、批量模式必须在同一视觉带中。
4. 主工作区：用 `WorkspaceGrid` 划分导航、列表、详情、图表或结果区域。
5. 内容表面：用 `Surface`/`Card`/`SettingsSection` 表达真实分组，禁止每一层都加边框。
6. 状态与反馈：持续状态用 `Alert`、`StatusBadge`、`Progress` 或固定状态条；短时成功信息使用现有 Toast；不得用突然出现的普通段落挤动主布局。

业务页面仍只能从 `components/ui.ts` 引入可见控件。若现有布局组件不足，应先在统一组件层新增纯展示型布局组件并进入 `components.css`，再由页面组合；不要给单页开 CSS 例外。

### 5.2 响应式规则

响应式只改变排版，不改变功能和窗口类型：

- 宽工作区（内容宽度 ≥1184px）：允许三段式工作区或侧栏 + 双列内容；工具栏尽量单行。
- 中等工作区（900–1183px）：侧栏保持 220–260px，其余内容单列；次要操作允许换行或进入现有“更多”菜单，但不得新造菜单功能。
- 紧凑工作区（<900px）：双栏切为上下布局或窄侧栏；表单双列切单列；工具栏可换两行；任何主按钮和危险操作仍可见。
- 数据表和行情指标不压缩到不可读：优先让局部内容区滚动，页面根不得出现无控制的横向滚动。
- 主内容滚动与标题/关键工具栏滚动分离；标题栏永远固定，批量选择栏、确认按钮等高频操作在长列表时应保持可达。
- 透明/不可缩放窗口（Pet、TrayMenu、MusicVisualizer、AgentConfirm）不套用普通模块断点，只按其固定窗口建立内部安全区。

### 5.3 视觉差异化原则

新 UI 不照抄旧 APP，也不能让所有页面长成同一个设置表单：

- 媒体页面突出封面、标题、时长、进度和播放动作，使用“媒体工作区”节奏。
- 远程视频突出 URL 输入、解析过程、任务进度和结果状态，使用“任务流水线”节奏。
- 行情页面突出数字、涨跌、更新时间和图表，使用“数据看板”节奏，但仍采用固定亮色主题。
- Timer 突出时间数字和单一主操作，减少外围装饰。
- Vault/Scripts 突出“对象列表—编辑详情”的生产力工具结构。
- AgentConfirm 突出风险和最终决策，避免普通信息与风险信息同权重。
- TrayMenu 追求快速扫描和大点击区；Showcase 追求规范查找和验收效率。
- 颜色只表达语义状态；禁止为了“科技感”引入黑色暗屏、霓虹大面积背景或页面自定义主题。

## 6. 统一控件复用映射

| 场景 | 必须优先复用 | 禁止做法 |
|---|---|---|
| 页面标题与动作 | `WindowTitleBar`、`PageHeader`、`Toolbar`、`ActionGroup` | 普通 `Container` 堆标题和按钮 |
| 主工作区 | `AppViewport`、`PageContent`/`MainRegion`、`WorkspaceGrid`、`Surface` | 页面私有 Grid CSS、固定像素硬撑 |
| 搜索筛选 | `SearchBox`、`SegmentedControl`、`Select`、`Tag` | Pressable 模拟所有筛选器 |
| 列表/选择 | `ListBox`、`ListBoxItem`、`Pressable`、`ScrollArea` | 无焦点态的可点击块 |
| 设置项 | `SettingsSection`、`SettingRow`、`FormField` | Label、说明、控件散落在多层 Container |
| 数据状态 | `Badge`、`StatusBadge`、`Alert`、`Progress`、`EmptyState`、`Loading`、`ErrorState` | 仅靠文字颜色或挤占布局的临时段落 |
| 数据表/详情 | `DataTable`、`DetailList`、`DetailRow`、`Card` | 手工拼表头、列错位 |
| 媒体 | `Avatar`/`MediaImage`、`VideoPlayer`、`MarketChart` | iframe、自制播放器按钮或替换媒体行为 |
| 对话与危险操作 | `Dialog`、`ConfirmDialog`、`Drawer`、`Menu`、`ContextMenuSurface` | 自制遮罩、普通按钮直接执行危险操作 |
| 小窗与透明层 | `PetBubbleSurface`、`TrayMenuSurface`、`TransparentStage`、`TransparentCanvas`、`MediaCanvas` | 添加不透明全窗背景或破坏命中区域 |

允许为本任务提炼纯 UI 复合组件，例如 `MediaWorkspaceHeader`、`SelectionToolbar`、`MetricGrid`、`CompactStatusBar`，但它们不得持有业务状态、发起 Core 请求或改变事件语义。

## 7. 逐页重排要求

### 7.1 PetPage / PetBubble

**现状问题**

- 桌宠画布、加载/错误文字、气泡和右键菜单共享同一层级，缺少透明窗口安全区和覆盖层优先级说明。
- 右键菜单功能很多，若保持单一长列表，会弱化模式、媒体、语音、工具、退出等功能组。
- 气泡正文、分页和关闭/继续操作空间有限，长文本时容易形成拥挤的小块。

**建议布局**

- `TransparentStage` 继续作为唯一根层，内部明确划分模型渲染层、气泡层、菜单层和诊断/错误层；四层通过统一 z-index 管理，但不增加全窗背景。
- 模型画布始终以可见内容居中，加载/资源错误用紧凑 `Alert` 或专用透明状态面板，不能改变画布几何和 alpha 命中计算。
- `PetBubble` 使用 `PetBubbleSurface`，正文为独立滚动/分页区，翻页与关闭操作固定在底部；气泡不得遮挡角色主要面部区域，靠窗口边缘时保持 12px 安全边距。
- 右键菜单按现有项目顺序和行为分视觉组：显示模式、当前模式参数、语音与对话、业务窗口入口、窗口/退出。只能增加分隔与小标题，不能改项目、事件或层级行为。

**响应式/窗口约束**

- 保持 560×760、最小 160×160、透明、无框、不可缩放、跳过任务栏、置顶和点击穿透逻辑。
- 小尺寸状态下优先保证模型和菜单可操作，气泡最大宽度受窗口限制，长文本继续使用现有分页，不缩小正文到 12px 以下。

**逐页验收**

- 图片、PNG 序列、Live2D 三种显示不改变播放、暂停、帧率、轮播与命中行为。
- 右键菜单全部现有入口可见可达，边缘打开时不出窗口。
- 气泡长文本、分页、实时对话和错误文本不遮挡菜单，不产生不透明窗底。

### 7.2 VideoLibraryPage

**现状问题**

- 当前侧栏同时堆放浏览、专辑、标签，主区又同时堆常用操作、搜索、排序、筛选和批量操作，视觉入口过多且缺乏主次。
- 动态窗口标题承担“当前位置 + 数量 + 搜索结果”三种职责，页面上下文不稳定。
- 视频列表虽然使用 `ListBoxItem`，但封面、状态、描述和主要播放行为没有形成清晰媒体层级；空列表和忙碌状态也不够稳定。
- 批量模式替换整段工具区，按钮多且易换行失控。

**建议布局**

- `WindowTitleBar` 固定为“视频库”；当前视图标题、视频数量和筛选摘要进入 `PageHeader`。
- 主区采用 240px 左导航 + 自适应内容区。左侧分“浏览、专辑、标签”三个有标题的滚动段，计数右对齐；新建按钮放对应段标题，不与列表项争抢空间。
- 内容区顶部建立固定工具带：第一行放导入文件、导入文件夹、字幕、依赖、选择；第二行放 `SearchBox`、排序 `SegmentedControl` 或等价现有控件、筛选 `Menu`。不得新增列表/网格切换状态。
- 列表项按“封面—标题与描述—状态—快捷信息”对齐，封面维持统一比例；单击/双击/选择行为严格保留当前实现。
- 批量模式使用固定 `SelectionToolbar`：左侧选中数量和全选/取消，右侧常用批量操作，低频项继续使用现有“更多”菜单；删除本地文件保持危险分组和确认。
- 底部状态变为固定 `CompactStatusBar`，忙碌、完成与错误不会推动列表高度。

**响应式规则**

- 默认 1480×820：侧栏 + 内容区；最小 1120×720 仍保持双栏，侧栏可压到 220px。
- 工具栏在最小宽度可分两行，但搜索框不得小于 240px；列表描述允许省略，标题和状态不得被挤掉。
- 专辑/标签自身滚动，页面标题和工具栏不随长列表消失。

**逐页验收**

- 浏览、专辑、标签、搜索、排序、筛选、单选/批量选择结果与改造前完全一致。
- 所有导入、重命名、标签、移动、收藏、备注、缩略图、删除记录、删除本地文件和依赖检测对话框保持原事件及文案语义。
- 0、1、100+ 视频以及超长标题/路径/标签下无错位、遮挡和横向页面滚动。

### 7.3 VideoPlayerPage

**现状问题**

- 当前页面只有标题栏和播放器，加载、错误、媒体区域三者没有稳定占位，状态切换时页面重排明显。
- 播放器在小窗口内缺少明确安全边距与错误展示层。

**建议布局与验收**

- 保持 `WindowTitleBar` + 单一沉浸式媒体舞台；播放器填充可用区，四周仅保留必要安全边距，不增加业务工具栏。
- 加载态、错误态覆盖同一个媒体舞台，不改变窗口高度；错误使用 `ErrorState`/`Alert`，但不得添加自动转码、外部播放器或重试兜底。
- 保持 720×480、最小 480×420 和可缩放窗口；原生 `<video>` 的自动播放、进度上报、格式错误处理全部不变。

### 7.4 VideoSubtitlesPage

**现状问题**

- 添加、递归导入、删除、刷新全部塞入标题栏，紧凑宽度下拥挤；字幕路径与名称缺少列对齐和选中信息。
- 空状态、读取状态、删除确认与列表之间层级弱。

**建议布局**

- 标题栏只保留窗口标题；下方 `Toolbar` 放添加、递归导入、刷新，危险删除靠右并随选中状态禁用。
- 字幕资源使用 `ListBox` 或两列 `DataTable`（名称/路径），选中态、键盘焦点和长路径截断必须明确；完整路径通过 Tooltip 或详情行可读。
- 读取中、空状态、删除确认分别使用 `Loading`、`EmptyState`、`ConfirmDialog`，不得改变导入/删除行为。

**验收**

- 保持 720×520、最小 560×420；最小宽度工具栏允许换行但不遮挡。
- 单文件导入、文件夹递归导入、刷新、选中和删除结果与原实现一致。

### 7.5 RemoteVideoPage

**现状问题**

- URL 输入、粘贴、清空、解析和站点/Cookie/数量状态挤在一个顶部块中，主动作不够突出。
- 三个 Tab 与诊断、站点配置、下载设置同处导航行，内容导航和全局工具混在一起。
- 解析结果的选择、格式、播放、下载信息密度高；下载记录的进度、错误和动作缺乏一致列结构。
- 设置和诊断 Drawer 内容较长，分组和底部动作不稳定。

**建议布局**

- 顶部建立“解析工作台”：URL 输入占主宽，解析链接为唯一 primary；粘贴、清空为次操作。站点识别、Cookie 来源和解析数量使用紧凑状态行。
- 第二层拆为左侧 `Tabs`（解析结果/下载记录/播放记录）和右侧页面工具（诊断、站点配置、下载设置），两组之间有明确间隔。

## 9. 全局验收标准

### 9.1 功能零变化核对

- 改造前后 `bridge.core.invoke` 的 route、payload 字段、timeout 和触发条件逐文件对比一致。
- `bridge.window.open/close`、`bridge.tray.action`、`bridge.pet.*`、`bridge.douyin.*`、媒体事件订阅和 DOM media 事件不变。
- 所有现有按钮、菜单项、右键、双击、键盘、选择、导入、播放、下载、删除、保存、恢复、复制入口数量一致。
- DTO、类型、共享常量、窗口注册、CoreHost、数据库和平台层零修改。

### 9.2 视觉与布局核对

- 每个普通窗口在默认尺寸和注册的最小尺寸各截图一次；透明/固定小窗按其实际运行尺寸截图。
- 不允许页面根横向滚动、文字压住按钮、Footer 被内容推出、弹层超窗、菜单出界、表头与行错列。
- 长中文/英文标题、长路径、长 URL、长 Cookie、长 JSON、长密钥、100+ 列表项、空数据、加载、错误、disabled、busy 均需覆盖。
- 标题、工具栏、侧栏、主区和状态区层级清晰；同页不得出现三个以上同权重 primary 操作。
- 焦点环、Tab 顺序、Enter/Space、Escape、aria-label、状态文本和非颜色提示可用。
- 主题选择按钮仍存在；全局搜索确认无深色模式、浅色模式、跟随系统模式、自动模式入口或 `prefers-color-scheme` 配色分支。

### 9.3 工程核对

- 页面只从 `components/ui.ts` 引入可见控件，不新增页面 CSS、内联视觉 style 和裸控件。
- 运行 UI 边界检查、ESLint、TypeScript、全部测试、生产构建和打包 Showcase 截图检查。
- 检查工作区差异，只提交本任务页面、必要统一组件/样式、Showcase 和对应测试；不得覆盖用户已有脏工作区修改。
- 本任务不是发布任务，不修改版本号，不提交、不部署，除非另有明确指令。

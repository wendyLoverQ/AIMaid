## 3. 新项目中已经确认的问题

### 3.1 历史定位代码把 `getBounds()` 的返回尺寸继续作为下一次输入

问题发生时，`apps/desktop/src/main/windows/window-manager.ts` 的宠物关联窗口定位代码是：

```ts
const targetBounds = positionWindowNearPet(
  window.getBounds(),
  petBounds,
  workArea,
  petWindowAlignment(kind, context.petDisplayMode)
)
window.setBounds(targetBounds, false)
```

同一文件的 `updatePetVisualBounds` 会在每次人物 item bounds 更新后，对已经登记的宠物关联窗口再次执行 `positionWindow`。

完整调用链在源码中可以直接追到：

```text
PetItemInteractionController.queueVisualBoundsReport
  -> preload pet.reportVisualBounds
  -> PetWindowManager.reportVisualBounds
  -> PetWindowManager.updateAbsoluteVisualBounds
  -> WindowManager.updatePetVisualBounds
  -> WindowManager.positionWindow
  -> BrowserWindow.setBounds
```

对应文件：

- `apps/desktop/src/renderer/shared/pet-item-interaction-controller.ts`
- `apps/desktop/src/preload/index.ts`
- `apps/desktop/src/main/windows/pet-window-manager.ts`
- `apps/desktop/src/main/windows/window-manager.ts`

### 3.3 可见聊天面板被后置 CSS 规则放大

Git `HEAD` 中，`apps/desktop/src/renderer/components/components.css` 对同一个元素先后存在三条宽度规则：

```css
body.prompt-surface .ui-main-region > .ui-section {
  width: min(380px, calc(100vw - var(--space-8)));
}

body.prompt-surface .ui-main-region > .ui-section {
  width: min(46rem, calc(100vw - var(--space-8)));
}

@media (max-width: 44.9375rem) {
  body.prompt-surface .ui-main-region > .ui-section {
    width: calc(100vw - var(--space-4));
  }
}
```

三条选择器的目标相同，后出现的规则覆盖前面的 `380px`。聊天外窗宽度为 520px，满足 `max-width: 44.9375rem` 媒体条件，所以最终还会命中第三条规则。这与老项目固定 `380` 的可见面板不一致。

## 4. 快捷键链路的当前源码事实

`apps/desktop/src/shared/system-settings.ts` 定义：

```ts
{ action: 'open-chat', defaultGesture: 'Ctrl+Shift+F', target: 'chat' }
```

`apps/desktop/src/main/services/system-settings-service.ts` 当前执行目标窗口动作时调用：

```ts
this.windows.toggle(definition.target, 'pet', {
  trigger: 'global-hotkey',
  petDisplayMode: this.presentation.snapshot().mode
})
```

`WindowManager.toggle` 的当前语义是：窗口已显示则 `hide()`；否则 `open()`。

因此，当前工作区源码中的快捷键目标行为已经写成“再次按下关闭”。但是本轮使用系统合成按键进行自动验证时，日志没有产生新的 `Window toggled visible/hidden` 记录。这个结果不能证明真实键盘按键失败，也不能证明成功，所以本文不把快捷键运行验证标记为通过。

日志中还存在一次较早的明确记录：`2026-07-22T16:48:17.398Z` 的启动过程中，`CommandOrControl+Shift+F` 注册失败。后续启动没有同样的失败记录。仅凭这两点不能确定用户当前按键对应的是哪一次进程实例。

## 5. Electron 官方文档能够确认的坐标规则

Electron 官方 `screen` 文档明确说明：

- `screen.getCursorScreenPoint()` 返回 DIP 坐标，不是物理屏幕点。
- `screen.screenToDipPoint(point)` 把物理屏幕点转换为 DIP，并按该物理点所在显示器执行 DPI 缩放。
- `screen.screenToDipRect(null, rect)` 会按离该矩形最近的显示器进行物理矩形到 DIP 矩形转换。

官方文档：<https://www.electronjs.org/docs/latest/api/screen>

Electron 官方 `BrowserWindow` 文档明确说明：

- `setBounds` 会移动并调整窗口到传入的 bounds。
- `getBounds` 返回窗口当前 bounds。

官方文档：<https://www.electronjs.org/docs/latest/api/browser-window#winsetboundsbounds-animate>

这些文档支持“物理桌面坐标必须先按目标显示器转换为 DIP，再交给 Electron 窗口 API”，但官方文档本身不证明本项目具体换算代码正确；项目正确性仍需要三屏运行数据证明。

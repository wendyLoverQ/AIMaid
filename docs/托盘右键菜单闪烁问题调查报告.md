# 托盘右键菜单闪烁问题调查报告

## 1. 当前结论

截至 2026-07-23，托盘右键菜单“闪两次”仍可复现。提交 `7a588d4` 引入的 100ms 右键延迟没有解决问题，并且在菜单已经显示时会放大一次明确的“先隐藏、再显示”过程。

目前证据表明，这不是单一问题，而是两个不同的显示路径：

1. **菜单已经可见时再次右键托盘**：菜单先因 `blur` 被隐藏，100ms 后右键回调又将其显示。这条关闭再打开的链路已经由日志和代码共同确认。
2. **菜单首次创建并显示时**：窗口只执行了一次 `show`，但显示后 13ms 内执行了两次 `tray.resize`。这会造成可见窗口连续改变边界，是首次打开产生两阶段画面的首要嫌疑，但现有证据还不足以断言它是全部闪烁的唯一原因。

因此，先前“Electron 一次右键触发两次应用回调”的判断已被运行日志否定，不能再按这个方向修改。

## 2. 复现环境

- 平台：Windows
- 桌面端：Electron 43.1.1
- 验证版本：包含 `7a588d4 fix(desktop): defer Windows tray right-click menu`
- 相关历史修改：
  - `bbfd9ac`：全窗口关闭 `backgroundThrottling`
  - `4fe79b2`：删除旧的重复 `show()` / `focus()`，并为加载完成显示增加一次性保护
  - `7a588d4`：右键事件增加 100ms 延迟与定时器合并

## 3. 运行日志证据

### 3.1 首次右键：只显示一次，但显示后连续 resize 两次

| UTC 时间 | 事件 |
| --- | --- |
| 17:50:08.834 | 创建 `tray-menu` 窗口 |
| 17:50:08.938 | 窗口显示 |
| 17:50:08.943 | 窗口获得焦点 |
| 17:50:08.946 | 第一次 `tray.resize` 开始 |
| 17:50:08.950 | 第一次 `tray.resize` 完成 |
| 17:50:08.959 | 第二次 `tray.resize` 开始 |
| 17:50:08.962 | 第二次 `tray.resize` 完成 |
| 17:50:10.057 | 窗口因失焦隐藏 |

关键事实：这一轮只有一条 `Window shown`，所以不是主进程把窗口 `show()` 了两次；但窗口已经可见后，renderer 又连续两次要求主进程修改窗口高度。

### 3.2 菜单可见时再次右键：确认存在 hide → 延迟 → show

日志中多次出现同样的时序：

| 隐藏时间 | 再次显示时间 | 间隔 |
| --- | --- | --- |
| 17:50:14.325 | 17:50:14.471 | 146ms |
| 17:50:14.989 | 17:50:15.119 | 130ms |
| 17:50:15.589 | 17:50:15.734 | 145ms |
| 17:50:16.182 | 17:50:16.316 | 134ms |
| 17:50:16.768 | 17:50:16.911 | 143ms |

这些间隔与当前 100ms 延迟以及事件处理开销吻合。每次只有一条 `Existing window opened` 和一条 `Window shown`，说明定时器确实合并了应用层回调，但它没有阻止窗口先被 `blur` 处理器隐藏。

## 4. 代码执行路径

### 4.1 右键被固定延迟 100ms

文件：[`apps/desktop/src/main/services/tray-controller.ts`](../apps/desktop/src/main/services/tray-controller.ts)

```ts
const RIGHT_CLICK_SETTLE_DELAY_MS = 100

private readonly queueRightClickMenu = (): void => {
  if (this.pendingRightClick !== undefined) clearTimeout(this.pendingRightClick)
  this.pendingRightClick = setTimeout(() => {
    this.pendingRightClick = undefined
    this.showMenu()
  }, RIGHT_CLICK_SETTLE_DELAY_MS)
}
```

这段代码只解决“短时间内收到多个应用层回调”的假设。实际日志显示每轮本来就只打开一次，因此它没有命中真实问题。

### 4.2 菜单一失焦就立即隐藏

文件：[`apps/desktop/src/main/windows/window-manager.ts`](../apps/desktop/src/main/windows/window-manager.ts)

```ts
if (kind === 'tray-menu') window.on('blur', () => window.hide())
```

当菜单已经可见并拥有焦点时，再次点击托盘会令它失焦，以上处理器立即隐藏窗口。随后延迟的 `right-click` 处理才调用 `showMenu()`，形成明确的：

```text
菜单可见
  → 点击托盘导致 blur
  → window.hide()
  → 等待 100ms
  → showMenu()
  → existing.show() + existing.focus()
```

这就是“菜单可见时再次右键”场景下已经确认的闪烁机制。

### 4.3 首次打开先显示，再由 renderer 改变窗口尺寸

文件：[`apps/desktop/src/main/windows/window-manager.ts`](../apps/desktop/src/main/windows/window-manager.ts)

```ts
window.once('ready-to-show', showLoadedWindow)
window.webContents.once('did-finish-load', showLoadedWindow)
```

`showLoadedWindow` 有一次性保护，所以这里只会显示一次。但是“页面加载完成”不等于“异步数据、主题和最终内容高度已经稳定”。

文件：[`apps/desktop/src/renderer/pages/system/TrayMenuPage.tsx`](../apps/desktop/src/renderer/pages/system/TrayMenuPage.tsx)

```ts
const observer = new ResizeObserver(report)
observer.observe(shell)
report()
```

`report()` 会立即上报一次高度，`ResizeObserver` 又可能在内容或样式改变后再次上报。页面还会异步读取音频状态和当前音乐，因此首次渲染后的内容高度可能再次变化。

文件：[`apps/desktop/src/main/ipc/ipc-router.ts`](../apps/desktop/src/main/ipc/ipc-router.ts)

```ts
window.setBounds({
  x: bounds.x,
  y: bottom - height,
  width: bounds.width,
  height
}, false)
```

当前 IPC 会直接修改已经显示的窗口边界。首次打开日志中的两次 `tray.resize` 与这条代码完全对应。

### 4.4 主题也在首次绘制后应用

文件：[`apps/desktop/src/renderer/theme/GlobalUiRoot.tsx`](../apps/desktop/src/renderer/theme/GlobalUiRoot.tsx)

```ts
useEffect(() => {
  const apply = (): void => applyStoredAppearance()
  apply()
  const unsubscribe = subscribeAppearance(apply)
  return unsubscribe
}, [])
```

主题在 React 首次提交后的 `useEffect` 中应用，理论上也可能造成一次可见样式变化。它目前只是候选因素，日志还没有证明它直接造成闪烁。

## 5. 已证实与已排除

### 已证实

- 每次右键只有一次应用层窗口打开记录，不存在两次 `showMenu()` 的日志证据。
- 100ms 延迟已经生效，但没有消除闪烁。
- 菜单可见时再次点击托盘，会出现 `blur → hide → 延迟 → show`。
- 首次创建窗口时，窗口显示后连续发生两次 `tray.resize`。
- `backgroundThrottling: false` 不能解决本问题。

### 已排除

- **一次 Windows 右键产生两次 JS `right-click` 回调**：与当前日志不符。
- **旧代码重复调用 `menu.show()` / `menu.focus()`**：该路径已在历史提交中删除，当前日志也只有一次显示。
- **把延迟从 100ms 继续加大**：只会扩大菜单隐藏与再次显示之间的空档。

### 尚未证实

- 两次 `tray.resize` 是否就是首次打开闪两次的唯一原因。
- 异步音乐状态、主题应用分别造成了哪一次高度或画面变化。
- 透明、无边框、置顶窗口在 Windows 合成器中的绘制是否还有独立影响。必须先消除应用自身的显示后 resize 和 hide/show，才能判断这一层。

## 6. 正确的下一步验证顺序

下一次修改不应继续猜测或增加延迟，应按以下顺序形成可证伪的证据链：

1. 给每次托盘交互分配 invocation ID，记录事件来源（左键/右键）、窗口操作前后的 `isVisible()`、`isFocused()` 和 `getBounds()`。
2. 在 `tray.resize` 日志中记录 renderer 请求高度、实际应用高度、修改前后 bounds，以及对应 invocation ID。
3. 将首次打开改成“隐藏创建 → renderer 明确上报 `tray.ready` 和最终高度 → 隐藏状态下定位/设尺寸 → 只显示一次”，验证显示后是否还能观察到 bounds 变化。
4. 对菜单已显示时的托盘右键建立由控制器管理的交互状态，避免 `blur` 在本次托盘事件结论产生前直接隐藏窗口；不能再用固定超时模拟事件边界。
5. 分别录制“首次从未创建状态打开”和“菜单可见时再次右键”的 60fps 视频，逐帧对齐窗口日志。

验收标准：

- 首次打开只有一次 `show`，且 `show` 后没有窗口边界修改。
- 菜单可见时再次右键，不出现 `hide → show` 对。
- 菜单隐藏时每次右键只显示一次，位置和最终高度在首帧即稳定。
- 连续快速右键不会残留定时器，也不会产生延迟重开。

## 7. 明确不要再做的修改

- 不再增加或调整固定延迟来“消抖”。
- 不在没有交互语义设计的情况下全局移除 `blur` 隐藏。
- 不再把问题归因于 Electron 重复派发，除非新增日志真的记录到两个回调。
- 不再调整 `backgroundThrottling` 作为此问题的修复手段。
- 不在未消除应用层二阶段尺寸变化前，把问题直接归咎于 Windows 透明窗口合成。

## 9. 最终判断

当前修复失败的直接原因是：**它试图消除并不存在的“双回调”，却没有处理真实存在的窗口生命周期冲突。**

- 菜单已经可见时，真实冲突是 `blur` 立即隐藏与延迟右键重新显示。
- 菜单首次打开时，真实风险是窗口先显示，renderer 随后连续改变窗口尺寸和内容状态。

后续实现必须分别关闭这两条路径，并按上述验收标准用日志与逐帧画面验证，不能再用肉眼感觉或固定延迟判断成功。
